01 Lifecycle hooks — before & after

Jest gives you four hooks to run code around your tests — so you don't repeat setup and teardown logic in every single test.

Hook When it runs Typical use
beforeAll() Once before all tests in the file/block Start a DB connection, spin up a server, load a large fixture
afterAll() Once after all tests finish Close DB connection, delete temp files, stop the server
beforeEach() Before every individual test Reset state, seed fresh data, clear mocks so tests don't bleed into each other
afterEach() After every individual test Clean up side effects, restore mocks, remove test records
⚠ Common mistake
Using beforeAll to set up shared mutable state — if one test changes it, the next test inherits a broken state. Use beforeEach to get a fresh copy every time.
// hooks.test.js — full lifecycle example 
const { createUser, deleteUser, getUser } = require('./userService');

let testUserId;

// Runs ONCE before all tests in this file
beforeAll(async () => {
  await db.connect(); // open connection once
});

// Runs ONCE after all tests finish
afterAll(async () => {
  await db.disconnect(); // close connection once
});

// Runs before EVERY test — fresh state each time
beforeEach(async () => {
  testUserId = await createUser({ name: 'Test User', email: 'test@test.com' });
});

// Runs after EVERY test — clean up so tests don't interfere
afterEach(async () => {
  await deleteUser(testUserId);
});

describe('user service', () => {

  test('getUser returns the correct user', async () => {
    const user = await getUser(testUserId);
    expect(user.name).toBe('Test User');
  });

  test('getUser throws for unknown id', async () => {
    await expect(getUser('nonexistent')).rejects.toThrow('User not found');
  });

});
💡 Scoping tip
Hooks inside a describe() block only apply to tests inside that block. You can nest describe blocks with their own hooks for fine-grained control.
02 Mocking functions inside functions

A mock is a fake version of a function that you control completely. Instead of calling the real thing (sending an email, charging a card), you replace it with a pretend version that just records what happened.

Why mock?
The real function is slow, expensive, unpredictable, or has side effects (email, payment, file system) — you want to test your code logic, not the external service.
What can you verify?
Was the function called? How many times? With which arguments? What did it return? All without it actually doing anything real.
// emailService.js — the function our code depends on 
async function sendWelcomeEmail(email) {
  // sends a real email — we do NOT want this in tests
  await mailer.send({ to: email, subject: 'Welcome!' });
}

module.exports = { sendWelcomeEmail };
// userController.js — the function we want to test 
const { sendWelcomeEmail } = require('./emailService');

async function registerUser(userData) {
  const user = await db.save(userData);
  await sendWelcomeEmail(userData.email); // ← we want to mock this
  return user;
}

module.exports = { registerUser };
// userController.test.js 
// Tell Jest to auto-mock the entire module
jest.mock('./emailService');

const { registerUser }    = require('./userController');
const { sendWelcomeEmail } = require('./emailService');

beforeEach(() => {
  jest.clearAllMocks(); // reset call history before each test
});

describe('registerUser', () => {

  test('sends a welcome email after registration', async () => {
    await registerUser({ name: 'Sara', email: 'sara@test.com' });

    // Was sendWelcomeEmail called?
    expect(sendWelcomeEmail).toHaveBeenCalledTimes(1);

    // Was it called with the right email?
    expect(sendWelcomeEmail).toHaveBeenCalledWith('sara@test.com');
  });

  test('does not send email if registration fails', async () => {
    db.save.mockRejectedValueOnce(new Error('DB error'));

    await expect(registerUser({ name: 'X', email: 'x@test.com' }))
      .rejects.toThrow('DB error');

    expect(sendWelcomeEmail).not.toHaveBeenCalled();
  });

});
ℹ Key mock matchers
toHaveBeenCalled() · toHaveBeenCalledTimes(n) · toHaveBeenCalledWith(...args) · not.toHaveBeenCalled()
03 Mocking a database

Unit tests should never touch a real database. A DB connection is slow, requires setup, and leaves dirty data. Instead, mock the DB layer so your test controls exactly what data comes back.

💡 Pattern to follow
Keep your DB calls in a separate module (e.g. userRepository.js). Your service only calls the repository — making it trivial to mock the repository in tests without touching the service logic.
// userRepository.js — the DB layer 
const db = require('./db');

async function findUserById(id) {
  return db.query(`SELECT * FROM users WHERE id = ?`, [id]);
}

async function saveUser(data) {
  return db.query(`INSERT INTO users SET ?`, [data]);
}

module.exports = { findUserById, saveUser };
// userService.js — the business logic layer 
const repo = require('./userRepository');

async function getUserProfile(id) {
  const user = await repo.findUserById(id);
  if (!user) throw new Error('User not found');
  return { id: user.id, name: user.name, email: user.email };
}

module.exports = { getUserProfile };
// userService.test.js — mock the whole DB layer 
// Mock the entire repository module
jest.mock('./userRepository');

const { getUserProfile }  = require('./userService');
const repo                 = require('./userRepository');

describe('getUserProfile', () => {

  test('returns formatted user when found', async () => {
    // Tell the mock what to return
    repo.findUserById.mockResolvedValue({
      id: 1, name: 'Sara Ahmed', email: 'sara@test.com', password: 'secret'
    });

    const profile = await getUserProfile(1);

    expect(profile).toEqual({
      id: 1, name: 'Sara Ahmed', email: 'sara@test.com'
      // password is NOT exposed — our service strips it
    });
  });

  test('throws when user does not exist', async () => {
    repo.findUserById.mockResolvedValue(null); // DB returns nothing

    await expect(getUserProfile(999))
      .rejects.toThrow('User not found');
  });

  test('throws when DB itself fails', async () => {
    repo.findUserById.mockRejectedValue(new Error('Connection lost'));

    await expect(getUserProfile(1))
      .rejects.toThrow('Connection lost');
  });

});
mockResolvedValue(x)
Make the mock return a resolved Promise with value x. Use for simulating a successful DB response.
mockRejectedValue(err)
Make the mock return a rejected Promise. Use to simulate a DB error or network failure.
mockResolvedValueOnce(x)
Like mockResolvedValue but only for the next call. Useful when the same mock needs different responses in sequence.
mockReturnValue(x)
Synchronous version — makes the mock return x directly (not a Promise). Use for non-async fakes.

MongoDB with mongodb-memory-server

When your app uses Mongoose, jest.mock the model works for pure unit tests — but sometimes you want to test the full Mongoose query logic (validators, hooks, transforms) without a real server. mongodb-memory-server spins up a real MongoDB instance in memory. It starts in beforeAll, is wiped in afterEach, and is torn down in afterAll. No Docker. No external process.

📦 Install
npm install --save-dev mongodb-memory-server mongoose
Add "@mongodbjs/mongodb-memory-server": { "version": "latest" } in package.json under "config" if you want a pinned binary.
// models/User.js — a simple Mongoose model 
const mongoose = require('mongoose');

const userSchema = new mongoose.Schema({
  name:  { type: String, required: true },
  email: { type: String, required: true, unique: true },
  role:  { type: String, default: 'viewer' },
});

module.exports = mongoose.model('User', userSchema);
// services/userService.js — calls the model directly 
const User = require('../models/User');

async function getUserProfile(id) {
  const user = await User.findById(id).lean();
  if (!user) throw new Error('User not found');
  return { id: user._id, name: user.name, email: user.email };
}

async function registerUser(data) {
  const existing = await User.findOne({ email: data.email });
  if (existing) throw new Error('Email already in use');
  return User.create({ name: data.name, email: data.email });
}

module.exports = { getUserProfile, registerUser };
// tests/userService.mongo.test.js — full memory-server setup 
const { MongoMemoryServer } = require('mongodb-memory-server');
const mongoose              = require('mongoose');
const User                  = require('../models/User');
const { getUserProfile, registerUser } = require('../services/userService');

let mongoServer;

// ── Setup ──────────────────────────────────────────────────────────
beforeAll(async () => {
  mongoServer = await MongoMemoryServer.create();           // start in-memory instance
  await mongoose.connect(mongoServer.getUri());            // connect Mongoose to it
});

afterEach(async () => {
  await User.deleteMany({});                               // wipe between tests
});

afterAll(async () => {
  await mongoose.disconnect();
  await mongoServer.stop();                               // tear down the server
});

// ── Tests ──────────────────────────────────────────────────────────
describe('getUserProfile', () => {

  test('returns profile for an existing user', async () => {
    // Arrange — insert directly via the model
    const created = await User.create({ name: 'Sara Ahmed', email: 'sara@test.com' });

    // Act
    const profile = await getUserProfile(created._id);

    // Assert
    expect(profile.name).toBe('Sara Ahmed');
    expect(profile.email).toBe('sara@test.com');
  });

  test('throws when user does not exist', async () => {
    const fakeId = new mongoose.Types.ObjectId();
    await expect(getUserProfile(fakeId)).rejects.toThrow('User not found');
  });

});

describe('registerUser', () => {

  test('creates and returns a new user', async () => {
    const user = await registerUser({ name: 'Karim Nour', email: 'karim@test.com' });
    expect(user.email).toBe('karim@test.com');
    expect(user.role).toBe('viewer');                       // default from schema
  });

  test('throws on duplicate email', async () => {
    await User.create({ name: 'Karim Nour', email: 'karim@test.com' });
    await expect(registerUser({ name: 'Other', email: 'karim@test.com' }))
      .rejects.toThrow('Email already in use');
  });

  test('enforces required fields via schema validation', async () => {
    // name is required — Mongoose throws a ValidationError
    await expect(User.create({ email: 'no-name@test.com' }))
      .rejects.toThrow('Path `name` is required');
  });

});
⚠️ This is an integration test, not a unit test
mongodb-memory-server runs real Mongoose queries — so you're testing the full stack from service → model → DB engine. Use it when you specifically want to test schema validation, indexes, or query logic. For testing pure business logic, keep mocking the repository instead.
Unit test (mock the repo)
Fast, isolated, no I/O. Test only your service logic. The repo is a Jest mock — the DB is never touched.
Integration test (memory server)
Slower, but tests real queries, schema validators, unique indexes, and Mongoose middleware like pre('save').
04 Testing async functions

Real-world code is full of Promises — API calls, DB queries, timers. Jest needs to wait for them to finish. There are three ways to do it.

Pattern 1 — async / await (recommended)
The cleanest approach. Make the test function async, then await the call. Jest waits automatically.
test('fetches user successfully', async () => {
  const user = await fetchUser(1);
  expect(user.name).toBe('Sara');
});
Pattern 2 — return the Promise
Return the Promise from the test. Jest will wait for it to resolve or reject. Older style, still valid.
test('fetches user successfully', () => {
  return fetchUser(1).then(user => {
    expect(user.name).toBe('Sara');
  });
});
Testing for rejections (errors)
Two clean ways to assert that an async function throws.
// Option A — await + rejects
test('throws on invalid id', async () => {
  await expect(fetchUser(-1)).rejects.toThrow('Invalid id');
});

// Option B — try / catch
test('throws on invalid id', async () => {
  try {
    await fetchUser(-1);
  } catch (err) {
    expect(err.message).toBe('Invalid id');
  }
});
⚠ Common async mistake
Forgetting await — the test finishes before the Promise resolves, so it passes even when it should fail. Always use async/await or return when testing Promises.

Full realistic example — fetching from an API

// api.js 
const axios = require('axios');

async function fetchPost(id) {
  const { data } = await axios.get(`https://jsonplaceholder.typicode.com/posts/${id}`);
  return { id: data.id, title: data.title };
}

module.exports = { fetchPost };
// api.test.js — mock axios so no real HTTP request fires 
jest.mock('axios');

const axios          = require('axios');
const { fetchPost } = require('./api');

describe('fetchPost', () => {

  test('returns formatted post on success', async () => {
    axios.get.mockResolvedValue({
      data: { id: 1, title: 'Hello World', body: '...', userId: 1 }
    });

    const post = await fetchPost(1);

    expect(post).toEqual({ id: 1, title: 'Hello World' });
    expect(axios.get).toHaveBeenCalledWith(
      'https://jsonplaceholder.typicode.com/posts/1'
    );
  });

  test('throws when API call fails', async () => {
    axios.get.mockRejectedValue(new Error('Network Error'));

    await expect(fetchPost(1)).rejects.toThrow('Network Error');
  });

});
05 TDD vs BDD

Both are workflows, not libraries. They describe when you write tests and how you think about them — not which tool you use.

Aspect TDD — Test-Driven Development BDD — Behaviour-Driven Development
Core idea Write the test first, then write just enough code to make it pass. Write tests that describe behaviour in plain language, from the user's point of view.
Cycle Red → Green → Refactor. Write a failing test. Make it pass. Clean up. Given → When → Then. Given a context, when an action happens, then expect an outcome.
Who writes Developers Developers + QA + Product — tests become shared documentation
Language Technical — function names, input/output Plain English — readable by non-developers
Jest example test('returns 80 for 20% off 100', ...) it('should apply the discount to the price', ...)
Best for Library code, utilities, pure functions Features, user stories, APIs with clear business rules

TDD in practice — Red → Green → Refactor

🔴
Red — write a failing test first
The function doesn't exist yet. The test will fail — that's the point. It defines what you need to build.
🟢
Green — write the minimum code to pass
Don't over-engineer. Write just enough code to make the test pass — even if it's ugly.
🔵
Refactor — clean up with confidence
Improve the code — rename, restructure, optimise. The test is your safety net. If it still passes, you didn't break anything.
// BDD style — the describe/it structure reads like a spec
describe('Discount calculator', () => {

  describe('when a valid discount is applied', () => {
    it('should reduce the price by the discount percentage', () => {
      expect(calculateDiscount(100, 20)).toBe(80);
    });
  });

  describe('when the price is negative', () => {
    it('should throw an error', () => {
      expect(() => calculateDiscount(-1, 10)).toThrow();
    });
  });

});

// Note: in Jest, test() and it() are identical — it() is just BDD convention
ℹ In Jest, test() and it() are the same
Use test() for TDD-style names ("calculates discount correctly") and it() for BDD-style sentences ("should return 80 when 20% off 100"). The choice is a convention, not a technical difference.
